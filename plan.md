# yzh666.dev 个人网站开发计划

## 1. 项目概述

### 1.1 项目定位

`yzh666.dev` 是一个面向开发者、招聘者和技术同行的个人网站，主要承担以下职责：

- 展示有代表性的项目和个人工程能力
- 发布经过整理的技术文章
- 记录持续更新的学习笔记
- 介绍个人经历、关注方向和联系方式
- 提供网页简历与 PDF 简历下载

网站不是单纯的在线简历，也不是只有文章列表的博客，而是一个持续更新的个人技术档案。

### 1.2 核心目标

1. 访客能在短时间内了解“我是谁、会什么、做过什么”。
2. 项目页面能够说明问题、过程、技术决策和成果，而不仅是罗列技术栈。
3. Blog 与 Notes 有明确边界，便于长期维护。
4. 内容使用本地 Markdown/MDX 管理，新增内容不需要修改页面代码。
5. 页面加载快、移动端可用，并具备基础 SEO 能力。
6. 第一版保持静态站点架构，降低部署和维护成本。

### 1.3 目标用户

- 招聘者和面试官
- 开源项目使用者或协作者
- 对文章和笔记感兴趣的开发者
- 希望快速了解个人背景的同学和朋友

### 2.1 栏目职责

#### 首页

首页负责建立第一印象和引导访问，不承载所有完整内容。

建议内容顺序：

1. 姓名、身份和一句话介绍
2. Projects、GitHub 等主要操作入口
3. 3 个精选项目
4. 最近发布的 Blog 与 Notes
5. 当前关注或正在进行的事情
6. 联系方式和页脚

#### Projects

展示能够证明个人能力的项目。每个项目详情建议包含：

- 项目背景和要解决的问题
- 项目状态与时间
- 个人职责
- 技术方案和关键决策
- 功能截图或演示
- 遇到的问题及解决方式
- 最终结果与复盘
- 在线地址和代码仓库

#### Blog

用于发布经过系统整理、具有完整论述的技术文章。文章应尽量保持稳定，修改以修正或补充为主。

#### Notes

用于记录较短的学习内容、命令、踩坑记录、概念总结和阶段性理解。笔记允许持续修订，并显示最近更新时间。

```text
Blog  = 已经整理完成、适合完整阅读的内容
Notes = 正在积累、会持续更新的知识记录
```

#### About

介绍个人背景、技术兴趣、做事方式和网站之外的个人信息。避免直接复制简历内容。

#### Resume

提供结构清晰、适合浏览和打印的网页简历，并保留 PDF 下载入口。

## 3. 技术架构

### 3.1 推荐技术栈

| 范围 | 方案 | 用途 |
| --- | --- | --- |
| 主框架 | Astro + TypeScript | 静态页面生成、路由和组件组织 |
| 内容格式 | Markdown / MDX | 管理 Projects、Blog 和 Notes |
| 内容管理 | Astro Content Collections | 内容校验、查询和类型安全 |
| 样式 | Tailwind CSS 或原生 CSS 变量 | 页面布局和设计系统 |
| 交互组件 | Astro 组件优先，按需使用 React | 搜索、筛选等局部交互 |
| 图标 | Lucide | 导航和操作图标 |
| 代码高亮 | Astro/Shiki | Blog 与 Notes 的代码块 |
| 搜索 | Pagefind | 构建后的静态全文搜索 |
| 评论 | Giscus（可选） | 基于 GitHub Discussions 的评论 |
| 部署 | Cloudflare Pages 或 Vercel | 自动构建、HTTPS 和域名托管 |
| 统计 | Cloudflare Web Analytics（可选） | 隐私友好的访问统计 |

具体依赖版本在开始搭建时使用当时的稳定版本，不在计划文档中锁死。

### 3.2 架构原则

- **静态优先**：第一版不引入数据库和独立后端。
- **内容与界面分离**：文章和项目存放在内容目录，页面负责查询与呈现。
- **渐进增强**：核心内容在没有客户端 JavaScript 时也能访问。
- **组件适度复用**：复用导航、内容卡片、标签和文章布局，不提前设计复杂组件体系。
- **真实内容优先**：界面围绕真实项目和文章设计，避免用大量占位内容推动开发。
- **可迁移**：内容保持标准 Markdown/MDX，未来可以迁移到其他框架或 CMS。

### 3.3 页面渲染方式

```text
Markdown / MDX
      ↓
Content Collections 校验与读取
      ↓
Astro 页面在构建阶段生成静态 HTML
      ↓
Pagefind 建立静态搜索索引
      ↓
部署到 CDN
```

第一版不需要运行时 API。联系功能优先使用邮件链接或第三方表单服务，避免为了一个表单增加后端。

## 6. 视觉与交互方向

### 6.1 视觉定位：macOS 桌面隐喻

首页采用 **macOS 桌面隐喻** 作为核心视觉概念，将个人网站包装成一个可交互的操作系统桌面。这不是简单的视觉模仿，而是利用熟悉的桌面范式降低访客的认知负担，同时通过细节（启动动画、Dock、菜单栏）传递个人风格和技术能力。

整体设计分为两层：

- **桌面层（首页）**：macOS 桌面体验，含菜单栏、桌面快捷方式、Dock、系统组件、通知中心
- **内容层（子页面）**：纸白背景的工程档案风格，用于项目、文章、笔记等内容的阅读和浏览

### 6.2 桌面层色彩

```text
菜单栏：深色半透明玻璃态（#1e1e1e / 80% opacity + blur）
壁纸：抽象渐变背景
桌面图标：白色文字 + 投影，保证在各种壁纸上可读
Dock：浅色毛玻璃（#f0f0f0 / 70% opacity + blur）
系统组件：半透明卡片（#ffffff / 60% opacity + blur）
强调色：macOS 风格蓝（#007aff）
状态指示：绿色（#34c759）— 在线 / 可用
```

### 6.3 内容层色彩（子页面）

```text
背景：纸白（#f3f0e9）
正文：炭黑（#17232d）
次要文字：中性灰（#667078）
主强调色：钴蓝（#176783）
辅助强调色：少量橙红
```

最终颜色通过 CSS 变量统一管理，确保正文和交互控件满足可访问性对比度要求。

### 6.4 桌面层交互

桌面层通过状态机管理多个阶段：

```
[关机] → 点击 POWER ON → [启动中] → 进度条完成 → [桌面]
                                                      │
                                               点击关机按钮
                                                      ↓
                                              [关机中] → [关机]
```

**桌面元素**：

- **菜单栏**（顶部）：品牌标记、”Desktop” 标题、导航链接、时钟和状态指示器
- **桌面快捷方式**（中央网格）：6 个可点击图标，进入对应内容区域。入场时错开动画
- **系统组件**（左侧）：天气（Open-Meteo API）、站点运行时间、本地访问计数
- **Dock**（底部）：常用链接（Desktop、GitHub、Resume、Mail）+ 关机按钮。悬停放大效果 + 工具提示
- **通知中心**（右上角）：最新内容更新通知
- **右键菜单**：桌面空白区域右键触发，提供快捷操作

**桌面内窗口系统**：点击桌面图标或 Dock 图标可打开 macOS 风格的浮动窗口，窗口可拖拽、最小化、关闭，支持 z-index 管理。

### 6.5 内容层交互原则

- 桌面端导航始终清楚（菜单栏），移动端使用可滚动的水平导航
- 当前栏目具有明确的导航状态（`aria-current=”page”`）
- 标签和筛选不遮挡主要内容
- 动效集中在页面进入、链接反馈和列表筛选
- 尊重 `prefers-reduced-motion`（动画缩短至 1ms）
- 所有关键操作支持键盘使用
- 链接、按钮和静态标签在视觉上能够清楚区分

### 6.6 字体系统

```text
标题（Desktop 标识）：系统默认 monospace
UI 文字：Manrope（sans-serif）
正文阅读：Georgia / Songti SC（serif）
代码 / 技术标签：IBM Plex Mono（monospace）
```

### 6.7 响应式适配

- **桌面端（≥1024px）**：完整 macOS 桌面体验（菜单栏 + Dock + 桌面图标 + 系统组件）
- **平板（768-1023px）**：简化桌面（缩小图标、Dock 简化）
- **移动端（<768px）**：桌面降级为列表式导航，Dock 隐藏，桌面快捷方式改为卡片列表
- **短屏幕（<680px 高度）**：系统组件和顶部间距压缩

## 10. 部署与更新流程

### 10.1 部署流程

```text
本地开发
   ↓
提交到 GitHub
   ↓
Pull Request / 预览部署
   ↓
构建和检查通过
   ↓
合并到主分支
   ↓
Cloudflare Pages 或 Vercel 自动部署
   ↓
yzh666.dev 更新
```

### 10.2 日常内容更新

发布文章或项目时：

1. 在对应内容目录中新建 Markdown/MDX 文件。
2. 填写 frontmatter。
3. 添加正文和图片。
4. 本地预览并检查移动端排版。
5. 将 `draft` 设置为 `false`。
6. 提交代码，等待自动部署。
