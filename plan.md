# yzh666.dev 个人网站开发计划

## 1. 项目概述

### 1.1 项目定位

`yzh666.dev` 是一个面向开发者、招聘者和技术同行的个人网站，主要承担以下职责：

- 展示有代表性的项目和个人工程能力
- 发布经过整理的技术文章
- 记录持续更新的学习笔记
- 介绍个人经历、关注方向和联系方式
- 提供网页简历与 PDF 简历下载

网站不是单纯的在线简历，也不是只有文章列表的博客，而是一个持续更新的个人技术档案。

### 1.2 核心目标

1. 访客能在短时间内了解“我是谁、会什么、做过什么”。
2. 项目页面能够说明问题、过程、技术决策和成果，而不仅是罗列技术栈。
3. Blog 与 Notes 有明确边界，便于长期维护。
4. 内容使用本地 Markdown/MDX 管理，新增内容不需要修改页面代码。
5. 页面加载快、移动端可用，并具备基础 SEO 能力。
6. 第一版保持静态站点架构，降低部署和维护成本。

### 1.3 目标用户

- 招聘者和面试官
- 开源项目使用者或协作者
- 对文章和笔记感兴趣的开发者
- 希望快速了解个人背景的同学和朋友

### 2.1 栏目职责

#### 首页

首页负责建立第一印象和引导访问，不承载所有完整内容。

建议内容顺序：

1. 姓名、身份和一句话介绍
2. Projects、GitHub 等主要操作入口
3. 3 个精选项目
4. 最近发布的 Blog 与 Notes
5. 当前关注或正在进行的事情
6. 联系方式和页脚

#### Projects

展示能够证明个人能力的项目。每个项目详情建议包含：

- 项目背景和要解决的问题
- 项目状态与时间
- 个人职责
- 技术方案和关键决策
- 功能截图或演示
- 遇到的问题及解决方式
- 最终结果与复盘
- 在线地址和代码仓库

#### Blog

用于发布经过系统整理、具有完整论述的技术文章。文章应尽量保持稳定，修改以修正或补充为主。

#### Notes

用于记录较短的学习内容、命令、踩坑记录、概念总结和阶段性理解。笔记允许持续修订，并显示最近更新时间。

```text
Blog  = 已经整理完成、适合完整阅读的内容
Notes = 正在积累、会持续更新的知识记录
```

#### About

介绍个人背景、技术兴趣、做事方式和网站之外的个人信息。避免直接复制简历内容。

#### Resume

提供结构清晰、适合浏览和打印的网页简历，并保留 PDF 下载入口。

## 3. 技术架构

### 3.1 推荐技术栈

| 范围 | 方案 | 用途 |
| --- | --- | --- |
| 主框架 | Astro + TypeScript | 静态页面生成、路由和组件组织 |
| 内容格式 | Markdown / MDX | 管理 Projects、Blog 和 Notes |
| 内容管理 | Astro Content Collections | 内容校验、查询和类型安全 |
| 样式 | Tailwind CSS 或原生 CSS 变量 | 页面布局和设计系统 |
| 交互组件 | Astro 组件优先，按需使用 React | 搜索、筛选等局部交互 |
| 图标 | Lucide | 导航和操作图标 |
| 代码高亮 | Astro/Shiki | Blog 与 Notes 的代码块 |
| 搜索 | Pagefind | 构建后的静态全文搜索 |
| 评论 | Giscus（可选） | 基于 GitHub Discussions 的评论 |
| 部署 | Cloudflare Pages 或 Vercel | 自动构建、HTTPS 和域名托管 |
| 统计 | Cloudflare Web Analytics（可选） | 隐私友好的访问统计 |

具体依赖版本在开始搭建时使用当时的稳定版本，不在计划文档中锁死。

### 3.2 架构原则

- **静态优先**：第一版不引入数据库和独立后端。
- **内容与界面分离**：文章和项目存放在内容目录，页面负责查询与呈现。
- **渐进增强**：核心内容在没有客户端 JavaScript 时也能访问。
- **组件适度复用**：复用导航、内容卡片、标签和文章布局，不提前设计复杂组件体系。
- **真实内容优先**：界面围绕真实项目和文章设计，避免用大量占位内容推动开发。
- **可迁移**：内容保持标准 Markdown/MDX，未来可以迁移到其他框架或 CMS。

### 3.3 页面渲染方式

```text
Markdown / MDX
      ↓
Content Collections 校验与读取
      ↓
Astro 页面在构建阶段生成静态 HTML
      ↓
Pagefind 建立静态搜索索引
      ↓
部署到 CDN
```

第一版不需要运行时 API。联系功能优先使用邮件链接或第三方表单服务，避免为了一个表单增加后端。

## 6. 视觉与交互方向

### 6.1 视觉定位

整体采用“工程档案 + 独立技术杂志”的设计方向：

- 页面克制、清晰，突出真实作品与文字内容
- 使用网格、编号、日期和细分隔线建立档案感
- 使用具有编辑感的标题字体和易读的正文排版
- 项目以真实截图为主要视觉内容
- 不使用大面积紫色渐变、装饰光球和模板化卡片墙

### 6.2 初步色彩

```text
背景：纸白或接近白色
正文：炭黑
次要文字：中性灰
主强调色：钴蓝
辅助强调色：少量橙红
深色模式：第二阶段根据需要实现
```

最终颜色应通过 CSS 变量统一管理，并确保正文和交互控件满足可访问性对比度要求。

### 6.3 页面识别系统

每类内容使用稳定的类型标识：

```text
PROJECT / 003
ARTICLE / 2026-06
NOTE / JAVA-CONCURRENCY
```

该标识可用于列表、详情页标题区和分享图片，形成网站的统一识别特征。

### 6.4 交互原则

- 桌面端导航始终清楚，移动端使用可访问的折叠菜单
- 当前栏目具有明确的导航状态
- 标签和筛选不遮挡主要内容
- 动效集中在页面进入、链接反馈和列表筛选
- 尊重 `prefers-reduced-motion`
- 所有关键操作支持键盘使用
- 链接、按钮和静态标签在视觉上能够清楚区分

## 7. 开发流程

### 阶段 0：内容准备与范围确认

目标：避免开发出只有占位符的空站。

任务：

- [ ] 确定网站名称、个人显示名称和一句话介绍
- [ ] 整理个人头像、GitHub、邮箱和其他公开链接
- [ ] 选择首批 3 个项目并准备截图
- [ ] 准备 1 篇 Blog 和 3 条 Notes
- [ ] 准备简历数据和 PDF 文件
- [ ] 确定首版是否需要中英文切换、暗色模式和评论

阶段产物：

- 内容清单
- 项目素材
- 网站基础文案

### 阶段 1：工程初始化

目标：建立可运行、可构建、可部署的 Astro 项目。

任务：

- [ ] 初始化 Astro + TypeScript
- [ ] 配置代码格式化和基础检查
- [ ] 建立目录结构
- [ ] 配置全局样式和设计变量
- [ ] 建立 `BaseLayout`
- [ ] 设置站点 URL、favicon 和基础 metadata
- [ ] 创建基础 404 页面

验收标准：

- 本地开发服务器正常运行
- TypeScript 检查通过
- 生产构建成功
- 首页和 404 页面可访问

### 阶段 2：内容系统

目标：让 Projects、Blog 和 Notes 都由内容文件生成。

任务：

- [ ] 定义三个 Content Collections
- [ ] 添加字段校验和草稿字段
- [ ] 建立测试内容
- [ ] 实现内容排序、筛选和日期格式化
- [ ] 实现动态详情路由
- [ ] 配置 Markdown/MDX 代码高亮

验收标准：

- 新增 MDX 文件后能够自动生成详情页
- 字段错误能在构建阶段被发现
- 草稿不会出现在生产构建中
- 内容列表按日期正确排序

### 阶段 3：全局界面与首页

目标：完成网站视觉基线和主要访问入口。

任务：

- [ ] 开发 Header、Footer 和移动端导航
- [ ] 完成全局字体、颜色、间距和响应式规则
- [ ] 开发首页身份区
- [ ] 展示精选项目
- [ ] 展示最近 Blog 与 Notes
- [ ] 添加当前动态和联系方式
- [ ] 完成页面进入和链接反馈动效

验收标准：

- 首页能够快速说明个人定位
- 主要内容在首屏或首轮滚动内出现
- 桌面端和移动端无内容重叠或横向滚动
- 关闭 JavaScript 后核心内容仍可访问

### 阶段 4：核心栏目

目标：完成网站第一版的全部主要页面。

任务：

- [ ] Projects 列表页与项目详情页
- [ ] Blog 列表页与文章详情页
- [ ] Notes 列表页与笔记详情页
- [ ] About 页面
- [ ] Resume 网页版和打印样式
- [ ] PDF 简历下载
- [ ] 上一篇/下一篇内容导航

验收标准：

- 所有一级导航页面均可访问
- 详情页 metadata、标题和内容正确
- 长文章在桌面端和移动端均易于阅读
- Resume 可以正常打印或导出 PDF

### 阶段 5：发现与分发

目标：让内容更容易被找到和订阅。

任务：

- [ ] 标签筛选与标签聚合页
- [ ] Pagefind 全文搜索
- [ ] 自动生成 RSS
- [ ] 自动生成 sitemap
- [ ] 配置 canonical URL
- [ ] 配置 Open Graph 和社交分享图片
- [ ] 添加结构化数据

验收标准：

- 可以通过标题、摘要和正文搜索内容
- RSS 和 sitemap 能被正常解析
- 分享链接时显示正确的标题、摘要和封面
- 页面具有独立且准确的 SEO 信息

### 阶段 6：测试、优化与上线

目标：完成正式发布。

任务：

- [ ] 检查桌面端、平板和手机布局
- [ ] 检查键盘导航、焦点状态和语义结构
- [ ] 压缩并转换图片格式
- [ ] 检查失效链接
- [ ] 运行构建、类型检查和自动化测试
- [ ] 使用 Lighthouse 检查性能与可访问性
- [ ] 部署预览环境
- [ ] 绑定 `yzh666.dev`
- [ ] 配置 HTTPS 和生产环境变量
- [ ] 执行上线后的页面与链接冒烟测试

验收标准：

- 生产构建无错误
- 核心页面无明显布局偏移
- 所有内部链接和主要外部链接有效
- 移动端主要流程可正常使用
- Lighthouse 的性能、SEO、最佳实践和可访问性无严重问题

## 10. 部署与更新流程

### 10.1 部署流程

```text
本地开发
   ↓
提交到 GitHub
   ↓
Pull Request / 预览部署
   ↓
构建和检查通过
   ↓
合并到主分支
   ↓
Cloudflare Pages 或 Vercel 自动部署
   ↓
yzh666.dev 更新
```

### 10.2 日常内容更新

发布文章或项目时：

1. 在对应内容目录中新建 Markdown/MDX 文件。
2. 填写 frontmatter。
3. 添加正文和图片。
4. 本地预览并检查移动端排版。
5. 将 `draft` 设置为 `false`。
6. 提交代码，等待自动部署。
